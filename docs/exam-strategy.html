🧠 The 4-Step Technique
1

Read ALL answers before deciding

Never stop at the first answer that sounds right. AWS writes distractors that are "almost correct." Read every option before committing.

💡 The right answer is the one that best fits — not just one that fits.
2

Identify the KEY constraint in the question

Every AWS question has a hidden constraint — cost, performance, availability, security, or simplicity. Find it first. It eliminates most wrong answers instantly.

💡 "Most cost-effective" and "highly available" are mutually exclusive constraints. Don't mix them up.
3

Eliminate the obviously wrong answers (usually 2)

AWS almost always includes 2 clearly wrong answers and 2 plausible ones. Eliminate the obvious ones first. Now you have a 50/50 choice.

💡 Answers with "on-premises", "manually", or "single AZ" for high availability questions are almost always wrong.
4

Pick the most "AWS-native" solution

Between the final two options, AWS rewards managed services over custom code, serverless over EC2, and automation over manual processes.

💡 If one answer uses a managed AWS service and the other uses a custom solution — the managed service wins 90% of the time.
⚡ AWS Exam Keyword Decoder

Spot these words in a question and you instantly know what the answer must satisfy.

Cost

"Most cost-effective" / "cheapest"

Look for: Spot instances, Reserved instances, S3 over EBS, Serverless (Lambda, Fargate), data tiering (S3 IA, Glacier)

Availability

"Highly available" / "fault tolerant"

Look for: Multi-AZ, Auto Scaling, ELB, RDS Multi-AZ, Route 53 failover, S3 (11 nines)

Performance

"Low latency" / "fastest"

Look for: ElastiCache, CloudFront, DynamoDB DAX, Placement Groups, EBS io2, Global Accelerator

Security

"Secure" / "encrypted" / "compliant"

Look for: KMS, IAM roles (not users), VPC, Security Groups, AWS Macie, GuardDuty, CloudTrail

Simplicity

"Least operational overhead" / "simplest"

Look for: Managed services (RDS over EC2+DB), Serverless, AWS Backup, EventBridge, Step Functions

Scale

"Millions of requests" / "scale automatically"

Look for: SQS, SNS, Lambda, DynamoDB, Kinesis, API Gateway — NOT EC2 alone without Auto Scaling

A company runs a web application with unpredictable traffic spikes. They need a solution that is highly available and most cost-effective. The database stores session data. Which solution should a solutions architect recommend?
✅ Correct: B

Why eliminate A? Single AZ = not highly available. Eliminated first.
Why eliminate C? Single AZ RDS + single large instance — still not highly available.
Why not D? Plausible, but rewriting the entire app to Lambda is NOT "most cost-effective" or realistic here.
Why B? Auto Scaling handles spikes (cost-effective — you pay for what you use), ALB distributes traffic, ElastiCache handles session data, RDS Multi-AZ = high availability. ✅
A company needs to process 500,000 messages per day from IoT devices. The processing can be delayed by up to 1 hour. They want the least operational overhead solution. What should they use?
✅ Correct: C

Why eliminate A? "Custom software + EC2" = maximum operational overhead. Never the right answer when "least overhead" is the constraint.
Why eliminate D? "On-premises" + "replicated to AWS" — adds complexity, not reduces it.
Why not B? Kinesis + EC2 consumers still requires managing EC2 instances. Not truly managed.
Why C? SQS is fully managed (no servers). Lambda is serverless. DynamoDB is serverless. Zero infrastructure to manage = least operational overhead. ✅
⚠️ Common Traps to Avoid
🪤

"Almost correct" answers

AWS writes answers that are correct in isolation but wrong for the specific constraint asked. Always re-check against the KEY constraint you identified in Step 2.

🪤

Absolute language ("always", "never", "all")

Answers using absolute words are almost always wrong. AWS cloud is nuanced — very few things are absolute.

🪤

Over-engineered answers

AWS exams test best practices — not the most complex solution. If one answer has 5 steps and another has 2, the 2-step answer is usually right (when simplicity is the constraint).

🪤

Confusing "durable" with "available"

S3 is 99.999999999% durable (data won't be lost) but not 99.99% available (you might not always access it instantly). These are different. Know which the question asks for.

🪤

IAM users instead of IAM roles

Any answer using IAM users with long-term access keys for services/EC2/Lambda is almost always wrong. AWS best practice = IAM roles. Always.
