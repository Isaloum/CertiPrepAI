SAA-C03 Quick Reference

Key AWS services, patterns, and exam tips at a glance.

🔒 Secure Architectures (30%)
IAM RolesTemp credentials for services/cross-account; preferred over access keys
SCP vs IAM PolicySCPs = org-level guardrails; IAM = account-level permissions
KMSManaged keys; CMK for S3/EBS/RDS encryption at rest
SG vs NACLSG=stateful (instance); NACL=stateless (subnet) with allow+deny
VPC EndpointsGateway (S3/DynamoDB free); Interface (PrivateLink, ENI)
Secrets ManagerAuto-rotate DB/API credentials; use over SSM Parameter Store for secrets
GuardDutyML threat detection; no agents; enable org-wide via delegated admin
WAF + ShieldWAF=L7 rules (SQL/XSS); Shield Adv=DDoS + cost protection
🔄 Resilient Architectures (26%)
Multi-AZ RDSSynchronous standby; auto failover ~1-2 min; NOT for read scaling
Read ReplicasAsync replication; scale reads; can promote; cross-region supported
Auto ScalingMin/max/desired; target tracking preferred; scheduled for known patterns
ALB vs NLBALB=L7 path/host routing; NLB=L4 ultra-low latency, static IP
Route 53 PoliciesFailover, Weighted, Latency, Geolocation, Multivalue
SQSDecouple; Standard=at-least-once; FIFO=exactly-once ordered
S3 ReplicationCRR=cross-region DR; SRR=same-region compliance. Versioning required.
DR StrategiesBackup/Restore→Pilot Light→Warm Standby→Active-Active (cost↑, RTO↓)
⚡ High-Performing Architectures (24%)
CloudFrontCDN for static/dynamic; Origin Shield; Lambda@Edge; signed URLs
ElastiCacheRedis=persistence+HA+sorted sets; Memcached=simple cache+multi-thread
EBS Typesgp3=baseline 3000 IOPS; io2=up to 64K IOPS; st1=throughput HDD
EFS vs FSxEFS=Linux shared NFS; FSx for Windows=SMB; FSx Lustre=HPC
DynamoDBDAX=microsecond cache; GSI/LSI=query flexibility; Streams=change events
KinesisData Streams=real-time; Firehose=managed delivery to S3/Redshift
LambdaMax 15min; 10GB RAM; concurrency limit; provisioned concurrency=no cold start
S3 PerformanceTransfer Acceleration=edge upload; multipart upload for >100MB files
💰 Cost-Optimized Architectures (20%)
Spot InstancesUp to 90% off; interruptible; ideal for batch, fault-tolerant workloads
Reserved Instances1 or 3 year; Standard=no change; Convertible=can change family
Savings PlansCompute (EC2+Fargate+Lambda) vs EC2 Instance SP (larger discount)
S3 Storage ClassesStandard→IA→One Zone-IA→Glacier Instant→Flexible→Deep Archive
S3 LifecycleAutomate transitions between storage classes; expire old versions
ServerlessLambda+API Gateway+DynamoDB=no idle cost; pay per invocation
Trusted AdvisorIdentifies idle/underutilized resources, unattached EBS, unused EIPs
Data TransferInbound free; AZ-to-AZ costs money; use same-AZ for high-volume transfers
🏗️ Architecture Patterns
3-Tier Web AppALB → EC2 (Private) → RDS Multi-AZ. ALB in public subnets.
Serverless APIAPI Gateway → Lambda → DynamoDB. No servers, auto-scales.
Static WebsiteS3 + CloudFront + Route 53. OAC prevents direct S3 access.
Event-DrivenS3 Event → SQS/SNS → Lambda. Decouple producers from consumers.
Big Data PipelineKinesis → Lambda/Firehose → S3 → Athena/Redshift → QuickSight
Hybrid ConnectivityDirect Connect=dedicated; VPN=encrypted internet; DX+VPN=redundant
🚨 Disaster Recovery Tips
Backup – EBS/RDSEBS Snapshots, RDS automated backups → push to S3/S3-IA/Glacier + Lifecycle Policy + CRR
Backup – On-PremSnowball (bulk migration) or Storage Gateway (ongoing sync to S3)
HA – DNSRoute 53 DNS failover to migrate traffic across regions instantly
HA – ServicesRDS Multi-AZ, ElastiCache Multi-AZ, EFS, S3 — all multi-AZ by default
HA – NetworkSite-to-Site VPN as failover/backup for Direct Connect outages
Replication – DBRDS Cross-Region Read Replica, Aurora Global DB, on-prem → RDS replication
Replication – StorageStorage Gateway: continuous replication from on-prem to AWS
AutomationCloudFormation/Beanstalk recreate env; CloudWatch alarms reboot EC2; Lambda for custom logic
Chaos EngineeringNetflix "Simian Army" randomly terminates EC2 to validate resilience
💡 Exam Tips
HA vs Fault TolerantHA=minimal downtime (Multi-AZ); FT=zero downtime (active-active)
Scale reads?RDS Read Replica or ElastiCache, NOT Multi-AZ
Scale writes?DynamoDB, Aurora, or shard/partition your database
Shared storage?EFS for Linux multi-instance; FSx Windows for Windows workloads
No DB admin?Use managed service: RDS/Aurora, not EC2+MySQL
Cross-account accessIAM Role with trust policy + assume role, NOT share credentials
